# Agentic Engram (エージェンティック・エングラム)
**自律型ローカル記憶エコシステム アーキテクチャ設計書 (v4: 最終確定版)**

## 1. システムのコンセプトと基本要件

* **目的:** 人間の記憶プロセス（生の体験 → 睡眠時の情報整理 → 連想と論理的推論）を模倣し、AIエージェントに過去の開発知見を持たせる。
* **環境要件:** 外部API、クラウドDB、Docker、常駐型ローカルサーバーを一切必要としない、ファイルシステムベースの**完全ローカル・オーバーヘッドゼロ環境**。
* **究極の高耐久性 (Filebeat方式):** プロセスの強制終了（`Ctrl+C`）や長期間のセッション放置に耐えうるよう、ファイルシステム上の真実（更新日時 `mtime`）と「行数ポインタ」のみで状態を管理する。ステータスフラグ（completed等）は持たない。
* **拡張性を前提とした記憶構造:** 将来的に「意味の類似度（ベクトルDB）」と「論理的依存関係（グラフDB）」のハイブリッド検索（GraphRAG）を実現するデータ構造とする。

## 2. ディレクトリとコマンド体系

すべてのデータと設定は `~/.engram/` 配下に集約される。

```text
~/.engram/
├── short-term-memory/
│   ├── session_123_log.txt  <- Agent 1が現在書き込んでいる生ログ（短期記憶）
│   └── archive/             <- TTL（例: 7日間更新なし）を満たし退避されたログ
├── memory-db/               <- 完全ローカル・サーバーレスのDBストレージ群
│   ├── vector_store/        <- [V1実装] LanceDBのデータ（類似度検索用）
│   └── graph_store/         <- [将来実装] Kùzuのデータ（論理ネットワーク用）
├── scripts/
│   ├── ae-miner.py          <- [NEW] Agent 2の起動・カーソル管理・スキップ判定を担うハブ
│   ├── ae-save.py           <- 記憶保存用CLIツール
│   ├── ae-recall.py         <- 記憶検索用CLIツール
│   └── ae-console.py        <- 記憶管理用Webダッシュボード (Streamlit)
└── config/
    ├── AGENTS.md            <- Agent 1に読み込ませるSkill定義
    └── cursor.json          <- [NEW] 各ログファイルの「最後に読み込んだ行数」と「更新日時」を管理
```

## 3. コンポーネントと記憶のライフサイクル

### ① 生ログの垂れ流し（Agent 1: ストリーマー）
* **仕組み:** `script` コマンドを使用し、ターミナル上のすべてのやり取りを透過的に記録する。
* **運用:** 普段の開発開始時に以下のエイリアスを実行し、一切の要約やオーバーヘッドなしに生ログを `short-term-memory/` に追記し続ける。
  ```bash
  alias dev='script -q -a ~/.engram/short-term-memory/session_$(date +%Y%m%d_%H%M%S)_log.txt -c "claude"'
  ```

### ② 意味付け・抽出・コンフリクト解消（Agent 2: 砂金掘り / `ae-miner.py`）
* **実体:** cron等で定期実行されるPythonラッパー (`ae-miner.py`)。
* **堅牢な抽出ロジック:**
  1. `short-term-memory/` 内の各ファイルのOS上の更新日時（`mtime`）を確認。
  2. `cursor.json` の `last_checked_mtime` と比較し、**変更がなければスキップ**。
  3. 変更があれば、`last_read_line` から現在の最終行（EOF）までの差分を読み込み、Agent 2（LLM）に渡す。
  4. **[意味的スキップ判定]** Agent 2が「まだ作業中・格闘中」と判断した場合は `SKIP` を返し、**`last_read_line` は進めず**に `mtime` のみ更新する（次回、続きを含めて再読込するため）。
  5. 区切りがついていれば、自ら `ae-recall` を叩き、類似度閾値を元に「新規（INSERT）」か「上書き（UPDATE）」かを判断。JSON配列を `ae-save` に流し込む。
  6. 保存成功後、`last_read_line` を進め、`mtime` を更新する。
  7. **[TTLアーカイブ]** 最終更新日時から一定期間（例: 7日間）変更がないファイルは、安全なコールドログとして `archive/` へ移動する。

### ③ 記憶の永続化（`ae-save.py`）
* **実体:** Pythonスクリプト ＋ `LanceDB` ＋ `sentence-transformers`。
* **動作:** LLM等の重い依存を持たず一瞬で起動。Agent 2から受け取ったJSON配列をパースし、一回のモデルロードでLanceDBにバッチ保存（UPSERT）する。

### ④ 記憶の自律的な想起（`ae-recall.py`）
* **実体:** Pythonスクリプトによる超軽量な検索用CLI。
* **動作:** Agent 1のSkillとして登録。未知のエラーに直面した際、Agent 1が自発的にこのコマンドを叩く。`--format json|markdown` オプションを備える。

### ⑤ 記憶のメンテナンス（`ae-console.py`）
* **実体:** `streamlit` を利用したローカルWebアプリ。ブラウザ上でのデータ管理用。

## 4. データの「契約（Contract）」定義

### A. 状態管理 (`cursor.json`)
* ステータスフラグは持たず、OSの真実（mtime）と行数ポインタのみで管理する。
```json
{
  "session_20260313_1000_log.txt": {
    "last_read_line": 1542,
    "last_checked_mtime": 1741824000.0
  }
}
```

### B. Agent 2 ⇔ `ae-save` 間のI/Oスキーマ (標準入力JSON配列)
```json
[
  {
    "action": "SKIP", 
    "reason": "まだCORSエラーの原因を調査中であり解決していないため。"
  },
  {
    "action": "INSERT", // または UPDATE
    "target_id": null,  // UPDATE時は既存記憶のハッシュIDを指定
    "payload": {
      "event": "Ollama APIを叩くとCORSエラー",
      "context": "Next.jsのクライアントから直接フェッチ",
      "core_lessons": "Next.jsのRoute Handlerを経由させること",
      "category": "architecture", 
      "tags": ["Next.js", "Ollama", "CORS"],
      "related_files": ["app/api/chat/route.ts"],
      "session_id": "session_20260313_1000"
    },
    // V1ではJSON文字列として退避、V2でグラフDBへ展開
    "entities": ["Next.js", "Ollama API", "Route Handler"],
    "relations": [
      {"source": "Next.js", "target": "Route Handler", "type": "USES"}
    ]
  }
]
```

### C. LanceDB テーブルスキーマ (`ae-save.py` 内定義)
* `id` には `hash(session_id + eventの先頭20文字)` 等の決定論的ハッシュを用い、システム再実行時も安全に冪等性（UPSERT）を担保する。
```python
import pyarrow as pa
engram_schema = pa.schema([
    pa.field("id", pa.string()),
    pa.field("vector", pa.list_(pa.float32(), 384)), # 例: intfloat/multilingual-e5-small
    pa.field("event", pa.string()),
    pa.field("context", pa.string()),
    pa.field("core_lessons", pa.string()),
    pa.field("category", pa.string()),
    pa.field("tags", pa.list_(pa.string())),
    pa.field("related_files", pa.list_(pa.string())),
    pa.field("session_id", pa.string()),
    pa.field("timestamp", pa.timestamp('s')),
    pa.field("entities_json", pa.string()),  # V2 拡張用
    pa.field("relations_json", pa.string())  # V2 拡張用
])
```

## 5. 段階的な実装ステップ

* **Step 1: ベクトル記憶基盤の構築 (V1 - コア)**
  * `ae-save.py` および `ae-recall.py` の実装。
* **Step 2: 砂金掘りエンジンと耐久機構の実装**
  * `ae-miner.py` の実装（mtime監視、差分抽出、TTLアーカイブ処理）。
  * Agent 2用システムプロンプトの実装。
* **Step 3: 管理UIの作成**
  * `ae-console.py` (Streamlit) の実装。
* **Step 4: グラフネットワークへの拡張 (V2 - 完了)**
  * `Kùzu` を組み込み、`entities_json` / `relations_json` をネットワーク化。
  * ハイブリッド検索（ベクトル類似度 + グラフトラバーサル）を `ae-recall` に実装。
  * `ae-console` にグラフ統計 + エンティティ探索UIを追加。
* **Step 5: メモリ統合・スキル化 (V3 - 完了)**
  * LanceDBスキーマに `occurrence_count` (pa.int32()) を追加。メモリの出現回数を追跡し、MERGE時に合算する。
  * `consolidate.py`: 全メモリのベクトルからコサイン類似度を計算し、閾値（デフォルト0.90）以上のペアを連結成分アルゴリズムでクラスタリング。
  * LLMに各クラスタを提示し、MERGE（統合）/ KEEP（維持）/ SKILL（手順書に昇格）を判断させる。
  * MERGE: 元メモリを削除し、統合メモリを1件挿入。`occurrence_count` は元メモリの合計を継承。
  * SKILL: MERGE処理に加え、Markdown形式のスキルファイル（`~/.engram/skills/*.md`）を生成。`occurrence_count >= 3` の場合のみ選択肢として提示。
  * `ae-consolidate.py` CLI: `--dry-run`（プレビュー）、`--threshold`、`--llm`、`--model` オプションを提供。
  * 詳細設計: `docs/prds/v3_consolidation.md` を参照。
